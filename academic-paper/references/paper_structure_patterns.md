# Paper Structure Patterns — 9 Paper Structure Models

Used by `structure_architect_agent` and `intake_agent` to select the appropriate paper structure.

## Pattern 1: IMRaD (Introduction-Method-Results-Discussion)

**Best for**: Empirical research with original data collection and analysis
**Typical length**: 5,000-8,000 words
**Disciplines**: Sciences, Social Sciences, Education, Medicine

### Structure

```
1. Title Page
2. Abstract (150-250 words)
   Keywords (5-7)
3. Introduction
   3.1 Context and Background
   3.2 Problem Statement
   3.3 Research Gap
   3.4 Purpose and Research Questions
   3.5 Significance of the Study
4. Literature Review
   4.1 Theoretical Framework
   4.2 Theme 1: [related research]
   4.3 Theme 2: [related research]
   4.4 Theme 3: [related research]
   4.5 Summary and Conceptual Framework
5. Methodology
   5.1 Research Design
   5.2 Participants/Sample
   5.3 Data Collection
   5.4 Instruments/Measures
   5.5 Data Analysis
   5.6 Validity and Reliability
   5.7 Ethical Considerations
   [Optional: 5.8 Threat Model / Security Model — for privacy×finance papers using this pattern]
   [Optional: 5.9 Financial-Scenario Validation — for cross-domain papers]
6. Results/Findings
   6.1 Descriptive Statistics (if quantitative)
   6.2 Finding 1 (aligned with RQ1)
   6.3 Finding 2 (aligned with RQ2)
   6.4 Finding 3 (if applicable)
7. Discussion
   7.1 Summary of Findings
   7.2 Interpretation and Comparison with Literature
   7.3 Theoretical Implications
   7.4 Practical Implications
   7.5 Limitations
   7.6 Future Research Directions
8. Conclusion
9. References
10. Appendices (if applicable)
```

## Pattern 2: Thematic Literature Review

**Best for**: Synthesizing existing research, identifying gaps, proposing future directions
**Typical length**: 6,000-10,000 words
**Disciplines**: All (especially Social Sciences, Education, Humanities)

### Structure

```
1. Title Page
2. Abstract
   Keywords
3. Introduction
   3.1 Topic and Rationale
   3.2 Scope and Boundaries
   3.3 Review Methodology (search strategy, inclusion criteria)
   3.4 Organization of the Review
4. Theme 1: [descriptive title]
   4.1 Sub-theme A
   4.2 Sub-theme B
   4.3 Summary of Theme 1
5. Theme 2: [descriptive title]
   5.1 Sub-theme A
   5.2 Sub-theme B
   5.3 Summary of Theme 2
6. Theme 3: [descriptive title]
   6.1 Sub-theme A
   6.2 Sub-theme B
   6.3 Summary of Theme 3
7. Cross-Cutting Synthesis
   7.1 Convergent Findings
   7.2 Divergent Findings and Debates
   7.3 Methodological Observations
8. Research Gaps and Future Directions
   8.1 Identified Gaps
   8.2 Proposed Research Agenda
9. Conclusion
   9.1 Key Takeaways
   9.2 Implications for Practice/Policy
10. References
```

**Best for**: Developing, critiquing, or extending theoretical frameworks
**Typical length**: 5,000-8,000 words
**Disciplines**: Social Sciences, Philosophy, Education

### Structure

```
1. Title Page
2. Abstract
   Keywords
3. Introduction
   3.1 The Theoretical Problem
   3.2 Why This Theory Matters
   3.3 Purpose and Contribution
4. Background: The Theory in Context
   4.1 Origins and Development
   4.2 Key Concepts and Definitions
   4.3 Current State of the Theory
5. Critical Analysis
   5.1 Strengths of the Framework
   5.2 Limitations and Gaps
   5.3 Competing Perspectives

6. Proposed Extension/Revision
   6.1 New Dimensions or Modifications
   6.2 Conceptual Model

7. Applications and Implications

   7.3 Methodological Implications

## Pattern 4: Case Study

**Best for**: In-depth analysis of specific institutions, programs, events, or phenomena
**Typical length**: 4,000-7,000 words
**Disciplines**: Education, Business, Social Sciences, Medicine (clinical)

### Structure

```
1. Title Page
2. Abstract
   Keywords
3. Introduction
   3.1 Background and Rationale
   3.2 Case Selection Justification
   3.3 Research Questions
4. Literature Context
   4.1 Relevant Theoretical Framework
   4.2 Prior Research on Similar Cases
5. Case Description
   5.1 Context and Setting
   5.2 Key Actors and Stakeholders
   5.3 Timeline of Events
   5.4 Data Sources
6. Analysis
   6.1 Analysis Framework
   6.2 Theme/Finding 1
   6.3 Theme/Finding 2
   6.4 Theme/Finding 3
7. Discussion
   7.1 Cross-Theme Synthesis
   7.2 Comparison with Literature
   7.3 Theoretical Insights
   7.4 Transferability (not generalizability)
8. Lessons Learned and Recommendations
   8.1 For Practitioners
   8.2 For Policymakers
   8.3 For Researchers
9. Conclusion
10. References
11. Appendices (case documents, data)
```

## Pattern 5: Policy Brief

**Best for**: Evidence-based policy recommendations for decision-makers
**Typical length**: 2,000-4,000 words
**Disciplines**: Public Policy, Education, Health, Economics

### Structure

```
1. Title and Author Information
2. Executive Summary (200-300 words)
   Key recommendations (3-5 bullet points)
3. Background
   3.1 The Policy Problem
   3.2 Current Policy Context
   3.3 Why Action is Needed Now
4. Evidence Base
   4.1 Key Finding 1
   4.2 Key Finding 2
   4.3 Key Finding 3
   4.4 International Comparisons (if relevant)
5. Policy Options Analysis
   5.1 Option A: [description]
       - Pros / Cons / Cost / Feasibility
   5.2 Option B: [description]
       - Pros / Cons / Cost / Feasibility
   5.3 Option C: [description]
       - Pros / Cons / Cost / Feasibility
6. Recommendations
   6.1 Recommended Option with Justification
   6.2 Implementation Steps
   6.3 Timeline
   6.4 Success Metrics
7. Conclusion
8. References
9. Appendix: Methodology Note
```

## Pattern 6: Conference Paper

**Best for**: Presenting research in progress or completed studies at academic conferences
**Typical length**: 2,000-5,000 words (varies by conference)
**Disciplines**: All

### Structure (Short Paper)

```
1. Title, Authors, Affiliations
2. Abstract (100-200 words)
   Keywords
3. Introduction
   3.1 Problem and Motivation
   3.2 Research Question(s)
4. Related Work
   4.1 Key Prior Studies
   4.2 Gap This Paper Addresses
5. Approach / Methodology
   5.1 Design
   5.2 Data/Materials

6. Results (Preliminary or Complete)
   6.1 Key Findings
   6.2 Tables/Figures
7. Discussion

## Pattern Selection Guide

| If your paper... | Use Pattern |
|-----------------|-------------|
| Has original data collection | IMRaD |
| Synthesizes existing literature | Literature Review |
| Develops/critiques a theory | Theoretical |
| Analyzes specific cases in depth | Case Study |
| Recommends policy actions | Policy Brief |
| Presents at a conference | Conference Paper |
| Designs a novel IT artifact with kernel-theory contribution for IS audience | DSR-MISQ (Pattern 7) |
| Proposes a new algorithm/protocol with formal complexity/security analysis | Algorithmic-Efficiency (Pattern 8) |
| Builds an analytical economic model of a technology-adoption or information-asymmetry problem | Econ-Model (Pattern 9) |
| Proposes a privacy-tech construction applied to a financial-market friction | Systems+Economics (see structure_architect_agent.md Pattern 7) |

**Cross-domain note**: Patterns 7–9 and Systems+Economics are NOT mutually exclusive with Patterns 1–6. A DSR-MISQ paper may adopt the IMRaD spine with design-science-specific section insertions. An Algorithmic-Efficiency paper may use the Conference Paper shell with expanded theoretical-analysis and reproducibility sections. The pattern labels describe the paper's intellectual architecture, not a rigid template.

## Pattern 7: DSR-MISQ (Design Science Research for MIS Quarterly / ISR)

**Best for**: Papers that design a novel IT artifact (construct, model, method, or instantiation) and contribute design principles grounded in kernel theory. This is the dominant paradigm for privacy-computing × finance papers at MISQ and ISR.
**Typical length**: 9,000–12,000 words
**Disciplines**: Information Systems, Design Science
**Source**: `shared/references/privacy_finance_methodology_presets.md` Recipe 1

### Structure

```
1. Title Page
2. Structured Abstract (≤150 words for MISQ; ≤150 words for ISR)
   Keywords (5–7)
3. Introduction
   3.1 The Business Problem / Financial Friction (open with the problem, not the technology)
   3.2 Why Existing Solutions Fail (both practice and prior research)
   3.3 Research Objective and Design-Science Framing
   3.4 Kernel Theory / Theoretical Lens (explicitly named)
   3.5 Contribution Summary (design principles + instantiation + evaluation)
   [Literature is embedded within the Introduction narrative — MISQ/ISR do not expect a standalone "Literature Review"]
4. Theoretical Foundations and Related Work (compact; 1,500–2,000 words)
   4.1 Kernel Theory (the theoretical lens that explains why the artifact works)
   4.2 Prior Design Efforts (what artifacts exist, why they are insufficient)
   4.3 Privacy-Technology Stream (MPC / FHE / DP / FL / ZKP / TEE — positioned as design material, not as contribution)
   4.4 Financial-IS Stream (the financial mechanism or market friction being addressed)
   4.5 Synthesis Gap: Why No Existing Artifact Satisfies the Design Requirements
5. Design Requirements and Principles
   5.1 Meta-Requirements (derived from kernel theory + problem analysis)
   5.2 Design Principles (prescriptive statements: "DP1: The artifact should enable [capability] because [kernel-theory justification]")
   5.3 Mapping: Requirements → Principles → Kernel Theory
6. Artifact Design and Description
   6.1 Artifact Overview and Architecture
   6.2 Core Components and Their Kernel-Theory Grounding
   6.3 Design Evolution (how the artifact changed during development; key design decisions)
   6.4 Instantiation (concrete implementation details; technology stack)
   [For privacy×finance: protocol description goes here — but always as a design artifact, not as a standalone crypto contribution]
7. Evaluation
   7.1 Evaluation Strategy (method, criteria, kernel-theory traceability)
   7.2 Evaluation Episode 1: [dimension] — method, results, design-principle link
   7.3 Evaluation Episode 2: [dimension] — method, results, design-principle link
   7.4 Evaluation Episode 3: [dimension] — method, results, design-principle link
   7.5 Cross-Episode Synthesis (what the ensemble of episodes tells us about the design principles)
   [For privacy×finance: evaluation episodes cover protocol performance AND financial-outcome metrics AND managerial validity]
8. Discussion
   8.1 Design Principles Revisited (what the evaluation taught us about each DP)
   8.2 Theoretical Contributions (to kernel theory, to design theory)
   8.3 Contributions to Practice (for financial institutions, regulators, technology providers)
   8.4 Boundary Conditions (where the artifact does and does not apply)
   8.5 Limitations and Future Research
9. Conclusion
   9.1 Summary of the Design-Science Contribution
   9.2 Key Takeaways for IS Scholars and Practitioners

10. References (MISQ Author-Date format)
11. Appendices (protocol details, security proofs, additional evaluation data)
```

### Word Allocation (10,000-word example)
| Section | % | Words |
|---------|---|-------|
| Introduction | 12% | 1,200 |
| Foundations & Related Work | 15% | 1,500 |
| Design Requirements & Principles | 10% | 1,000 |
| Artifact Design | 22% | 2,200 |
| Evaluation | 25% | 2,500 |
| Discussion | 12% | 1,200 |
| Conclusion | 4% | 400 |

### DSR Quality Gates (MISQ/ISR)
| Check | Criterion |
|-------|-----------|
| Kernel theory explicitly named and applied | Mandatory — desk reject if missing |
| Design principles numbered (DP1, DP2, ...) and traceable to kernel theory | Mandatory |
| Evaluation episodes are multi-method (not single-experiment) | MISQ expects ≥ 2 distinct evaluation methods |
| Artifact described at the design-principle level, not the code level | The paper should teach design knowledge, not document software |
| Managerial/practical implications are first-class (not an afterthought) | ISR desk-rejects papers that end on technical notes |
| Contribution is to IS design theory, not to CS cryptography | The framing determines which journal is appropriate |

### Anti-Patterns (MISQ/ISR)
- "We propose a privacy-preserving system for [financial task]" without naming the kernel theory — reads as a CS systems paper
- Evaluation is a single benchmark experiment — insufficient for DSR standards
- Design principles are post-hoc rationalizations rather than a priori commitments — reviewers will detect this
- The artifact is described as a finished product with no discussion of design evolution or rejected alternatives — misses the design-science contribution

## Pattern 8: Algorithmic-Efficiency (Management Science / INFORMS Journal on Computing)

**Best for**: Papers whose primary contribution is a new algorithm or protocol with formal complexity, security, or privacy guarantees, validated through computational experiments. Appropriate for Management Science (if the algorithm enables a material economic mechanism) or INFORMS Journal on Computing (if the algorithmic contribution is the central claim).
**Typical length**: 8,000–12,000 words
**Disciplines**: Computer Science, Operations Research, Information Systems
**Source**: `shared/references/privacy_finance_methodology_presets.md` Recipe 3

### Structure

```
1. Title Page
2. Abstract (≤150 words for Management Science; ≤200 words for INFORMS JoC)
   Keywords
3. Introduction
   3.1 The Computational Problem (concrete task; why existing algorithms are insufficient)
   3.2 The Financial/Economic Motivation (why solving this problem matters)
   3.3 Our Approach and Intuition (high-level idea, not detailed construction)
   3.4 Contributions (formal statement: complexity bound, security guarantee, empirical result)
4. Preliminaries and Problem Statement
   4.1 Notation and Definitions
   4.2 System Model and Threat Model
   4.3 Formal Problem Definition
   4.4 Baseline Algorithms / Prior Protocols (with their known bounds)
5. Algorithm / Protocol Description
   5.1 Overview and Intuition
   5.2 Detailed Construction (may span multiple subsections)
   5.3 Correctness Argument (informal; formal proof may be in appendix)
   5.4 Illustrative Example (walk through a small instance)
6. Theoretical Analysis
   6.1 Complexity Analysis (time, communication, space — with formal bounds)
   6.2 Security Analysis (threat model → security claim → proof sketch or reference to appendix)
   6.3 Privacy Analysis (if DP: ε, δ bounds; if MPC: what is and isn't leaked)
   6.4 Comparison with Baselines (analytical comparison, not just empirical)
7. Experimental Evaluation
   7.1 Experimental Setup (hardware, datasets, parameter ranges)
   7.2 Baselines and Metrics (which baselines, which metrics, and WHY each is chosen)
   7.3 Experiment 1: [dimension] — hypothesis, setup, results, interpretation
   7.4 Experiment 2: [dimension]
   7.5 Experiment 3: [dimension]
   7.6 Sensitivity Analysis (how results vary with key parameters)
   7.7 Scalability Analysis (how the protocol degrades as n, m, or data size grows)
8. Discussion
   8.1 Summary of Results
   8.2 When Our Protocol Is (and Is Not) the Right Choice
   8.3 Managerial/Economic Implications (required for Management Science)
   8.4 Reproducibility Statement (required for INFORMS JoC)
   8.5 Limitations
9. Conclusion

10. References (INFORMS Author-Date format)
11. Appendix
    A. Full Security Proofs
    B. Additional Experimental Data
    C. Pseudocode (if not in main text)
```

### Word Allocation (10,000-word example)
| Section | % | Words |
|---------|---|-------|
| Introduction | 10% | 1,000 |
| Preliminaries | 12% | 1,200 |
| Algorithm Description | 20% | 2,000 |
| Theoretical Analysis | 18% | 1,800 |
| Experimental Evaluation | 22% | 2,200 |
| Discussion | 10% | 1,000 |
| Conclusion | 3% | 300 |
| Appendix | — | not counted |

### Quality Gates
| Check | Criterion |
|-------|-----------|
| Formal problem statement with precise notation | Mandatory |
| Threat model stated before protocol description | Mandatory — desk reject at crypto-literate venues |
| Complexity/security bounds are formal theorems (not empirical observations) | Required for theoretical contribution claim |
| Every symbol defined before first use | Mandatory |
| Experimental section answers specific hypotheses, not "we ran benchmarks" | Required |
| Reproducibility: code + data + parameter settings available | Mandatory for INFORMS JoC; recommended for Management Science |

## Pattern 9: Econ-Model (Management Science Microeconomics / Finance Track)

**Best for**: Papers that build an analytical economic model to study how privacy technologies affect market outcomes — adoption incentives, welfare implications, information-asymmetry resolution, or regulatory design. This is NOT a "write code and test it" paper; the contribution is the model's insights, not a computational artifact.
**Typical length**: 8,000–10,000 words
**Disciplines**: Economics, Information Systems (Economics of IS track), Finance
**Source**: `shared/references/privacy_finance_methodology_presets.md` Recipe 2

### Structure

```
1. Title Page
2. Abstract (≤150 words for Management Science)
   Keywords
3. Introduction
   3.1 The Economic Phenomenon (a real-world tension or market failure)
   3.2 Stylized Facts (empirical regularities the model must explain)
   3.3 Related Literature and Positioning (compact — what models exist, why they don't capture this phenomenon)
   3.4 Preview of the Model and Main Results
   3.5 Contribution to Theory and Practice
4. Model Setup
   4.1 Agents, Preferences, and Information Structure
   4.2 Technology Description (the privacy mechanism, parameterized)
   4.3 Timeline and Decision Sequence
   4.4 Equilibrium Concept (e.g., Perfect Bayesian Equilibrium, Nash, etc.)
5. Equilibrium Analysis
   5.1 Benchmark: No-Privacy-Technology Equilibrium
   5.2 Equilibrium under Privacy Technology Adoption
   5.3 Comparative Statics (how outcomes change with key parameters)
   5.4 Welfare Analysis (who gains, who loses, total welfare effect)
6. Extensions and Robustness
   6.1 Extension 1: [e.g., competing platforms, heterogeneous agents]
   6.2 Extension 2: [e.g., regulator mandate, subsidy/tax]
   6.3 Robustness to Alternative Assumptions
7. Implications
   7.1 Managerial Implications (for banks, platforms, data intermediaries)
   7.2 Regulatory and Policy Implications (for central banks, data-protection authorities)
   7.3 Testable Predictions (what empiricists should look for in data)
8. Conclusion
   8.1 Summary of the Model's Insights
   8.2 Limitations of the Modeling Approach
   8.3 Future Theoretical and Empirical Work

9. References (INFORMS Author-Date format)
10. Appendix
    A. Proofs of Propositions
    B. Additional Comparative Statics
```

### Quality Gates
| Check | Criterion |
|-------|-----------|
| Stylized facts anchor the model in observable reality | Mandatory — reviewers will ask "why should we believe this model?" |
| Every parameter has economic meaning (not a free variable with no interpretation) | Mandatory |
| Formal proofs for all propositions (may be in appendix) | Mandatory |
| Comparative statics produce directional, signable predictions | Required |
| Welfare analysis identifies winners and losers (not just "total surplus increases") | Expected at Management Science |
| The privacy technology is modeled with fidelity to what it actually guarantees | CS reviewers will check this |

### Anti-Patterns
- "Consider a market with N banks..." without citing data on how many banks actually exist — model lacks empirical grounding
- The privacy technology enters as a "black box parameter" with no connection to actual cryptographic guarantees — CS-literate reviewers will reject
- "The regulator should mandate X" without discussing the regulator's information constraints or political feasibility — policy implication without mechanism
- All comparative statics have the same sign — suggests the model is under-parameterized or the results are driven by functional-form assumptions
